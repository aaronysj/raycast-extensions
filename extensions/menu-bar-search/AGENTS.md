# AGENTS.md

Guidance for coding agents working on Menu Bar Search.

## Project Goal

Menu Bar Search is a Raycast extension for finding and opening macOS menu bar items from Raycast, especially when crowded menu bars or the MacBook notch hide icons that can no longer be clicked with the pointer.

The extension must stay within public macOS Accessibility APIs. Do not introduce private APIs, background downloads, or opaque checked-in binaries.

## Repository Map

- `src/index.tsx`: Raycast command UI and command actions.
- `src/helper-client.ts`: TypeScript adapter for the Swift helper. Use generated Raycast preference types from `raycast-env.d.ts`; do not duplicate preference types by hand.
- `src/menu-bar-opening.ts`: Raycast-side opening fallback logic, including System Events fallback.
- `src/menu-bar-presentation.ts`: UI labels, icons, and open hints.
- `src/menu-bar-types.ts`: shared TypeScript protocol types.
- `helper/menubarctl.swift`: Swift helper that owns Accessibility scanning, semantic categorization, open policy, debug snapshots, and self-tests.
- `scripts/generate-icon.swift`: icon generation script.
- `metadata/*.png`: Raycast Store screenshots. Only keep final Store-ready PNGs here.
- `assets/*.svg` and `assets/extension-icon.png`: committed static assets.

## Generated and Local Files

- `assets/menubarctl` is generated by `npm run build-helper`; keep it out of git.
- `.build/`, `dist/`, `node_modules/`, `.idea/`, and `.DS_Store` are local-only.
- `raycast-env.d.ts` is generated from `package.json` by Raycast and should be committed because the TypeScript project uses its generated `Preferences` namespace.
- Avoid adding internal planning files such as naming notes, publishing checklists, or raw screenshot sources to Store-facing commits.

## Common Commands

- `npm run dev`: build the helper and start Raycast development mode.
- `npm run build-helper`: compile `helper/menubarctl.swift` into `assets/menubarctl`.
- `npm run test:helper`: build the Swift helper and run helper self-tests.
- `npm run build`: build the helper, compile the Raycast extension, generate Raycast types, and type-check.
- `npm run lint`: run Raycast manifest, icon, metadata, ESLint, and Prettier checks.
- `npm run publish`: open a Raycast Store submission PR.

## Validation Before Finishing

Run these before handing work back when code or manifest behavior changes:

```bash
npm run lint
npm run build
npm run test:helper
```

`npm run build` writes to the local Raycast extension directory and may require permission outside a sandboxed environment.

## Store and Metadata Rules

- Store screenshots must be PNG files in `metadata/`.
- Each metadata PNG must be exactly `2000 x 1250`.
- Keep only final uploadable screenshots in `metadata/`; raw captures belong outside the Store-facing tree or should be removed before publishing.
- README and `package.json` should emphasize the MacBook notch/camera-housing pain point: opening hidden menu bar icons that cannot be clicked directly.
- For Raycast Store PRs, mention that the helper is built from source:

```text
This extension builds its macOS helper from helper/menubarctl.swift during npm run build. It uses public Accessibility APIs only and does not download external binaries.
```

## Implementation Notes

- When adding new features, prioritize tool stability over capability breadth. Prefer conservative behavior that avoids crashes, hangs, accidental clicks, or misleading success states.
- Think through edge cases before implementing: missing Accessibility permission, stale menu bar rows, hidden notch/camera-housing items, apps that expose incomplete AX actions, missing helper binaries, slow or unresponsive target apps, empty lists, unusual input methods, and menu bar geometry changes.
- Add or extend boundary-case tests when feature behavior changes. At minimum, update Swift helper self-tests for open-policy/catalog changes and keep TypeScript build/lint coverage passing for Raycast-side behavior.
- Preserve compatibility with visible items, hidden items, system extras, input methods, and generic third-party status items. A fix for one category should not silently regress another.
- The Swift helper owns semantic category detection and open policy. Keep UI logic from re-inferring categories.
- Element identity should prefer stable owner/title/category facts. Treat frame and source as hints, not durable identity.
- For items hidden behind the camera housing with no physical click point, Accessibility may be the only available open path.
- For visible generic app status items, System Events can be used as a fallback when helper opening fails.
- Raycast may show a cached Menu Bar Catalog for fast launch, but opening must always re-resolve against the current Accessibility tree. Cached rows are display hints, not authority.
- Generated app icons should use a persistent lazy cache. Prefer `appPath`/Raycast file icons when available, and do not read `NSRunningApplication.icon` when the cached PNG already exists.
- Do not make `shouldOpenWithSystemEventsFirst` and fallback eligibility the same predicate; that makes the fallback branch unreachable.
- If `package.json` preferences change, run `npm run build` so `raycast-env.d.ts` is regenerated, then keep TypeScript usage aligned with the generated `Preferences` types.

## Git Hygiene

- Check `git status --short --branch` before editing and before finishing.
- Do not revert user changes unless explicitly asked.
- Keep Store submission commits focused: source, assets, metadata, README, CHANGELOG, and manifest changes only.
- Do not commit generated helper binaries or local build caches.
